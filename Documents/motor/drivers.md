# 电机驱动

本节面向维护者说明各驱动的职责边界、协议模型、反馈模型、配置项和已知边界。这里不复述每一类协议字段编码，只说明驱动在子系统中的位置，以及修改时需要守住的契约。

## 共性

除了 `yinshi,la` 外，本文中的驱动都实现 `include/zephyr/drivers/motor.h`。

共同职责有三项：

- 把 `motor_setpoint_t` 转成协议侧目标。
- 把协议反馈帧映射成 `motor_status_t`。
- 维护与 `motor_link`、`motor_can_sched`、控制器的交界。

驱动之间的主要差异体现在三处：

- 电机是周期主动上报，还是请求后回复。
- 控制闭环是在主控控制器层，还是电机协议原生提供。
- 设备树属性是描述物理极限，还是描述运行时策略。

## DJI

### 职责

DJI 驱动面向标准帧 CAN 电机，负责：

- 聚合同一 TX ID 下最多四个电机的控制量。
- 解析电机周期状态上报。
- 在本地执行位置/速度控制器，再把结果映射为目标电流或扭矩。

它是典型的“周期上报型 + 本地控制型”驱动。

### 协议模型

DJI 控制帧按 TX ID 聚合发送，一个帧里并列承载同组电机的控制输出。驱动为每个 CAN 设备维护一个 `ctrl_struct`，记录：

- 某个 TX ID 对应哪些逻辑电机槽位。
- 某个聚合帧是否已经集齐本轮输出。
- `follow` 电机是否需要跟随主电机输出。

维护这里时要注意，DJI 的调度单元是“一个聚合帧”，不是“一个 motor 设备”。

### 反馈模型

反馈由电机周期上报。驱动在 RX 回调里完成：

- 角度、速度、扭矩和温度解包。
- 多圈累计角度更新。
- `motor_link_observe_periodic_report()`。

离线检测不是等待回复，而是等待状态帧超时。测试里对该模型有单独覆盖。

### 控制模型

支持三条主要路径：

- `VO + TORQUE`
  直接扭矩目标，不走控制器。
- `VO + SPEED`
  使用匹配的 `motor-controller,vo` 选择速度环。
- `PV + POSITION`
  使用本地控制器计算中间速度和目标扭矩。

`motor_setpoint_t.torque` 在位置和速度模式下可作为扭矩前馈；只有驱动明确支持叠加时才生效。

### 配置

`dji,motor` 主要属性包括：

- `id`
- `tx_id`
- `rx_id`
- `can_channel`
- `gear_ratio`
- `controllers`
- `is_m3508` / `is_m2006` / `is_gm6020`
- `is_dm_motor`
- `dm_i_max`
- `dm_torque_ratio`
- `minor_arc`
- `inverse`
- `follow`

其中：

- `minor_arc` 控制位置闭环采用短弧误差。
- `inverse` 翻转角度或扭矩方向。
- `follow` 让从电机复用主电机输出。
- `is_dm_motor`、`dm_i_max`、`dm_torque_ratio` 用于把输出映射到带 DM 语义的量纲。

### 边界

- 聚合发送和 `follow` 逻辑耦合较紧，改动时要防止不同槽位互相覆盖。
- 离线逻辑依赖周期状态帧，不应改成请求-回复风格。
- `status.enabled` 近似在线状态，不等同于 `requested_enabled`。

## DM

### 职责

DM 驱动面向请求-回复型 CAN 电机，负责：

- 在 MIT、PV、VO 三种模式下打包目标帧。
- 按控制模式同步电机内部运行模式和部分控制寄存器。
- 跟踪回复超时并维护在线状态。

### 协议模型

DM 协议是标准帧，主控发送控制帧后等待反馈。驱动使用 `motor_can_sched_send_reply()` 提交周期控制请求，并在超时时累计丢失回复计数。

模式切换时会写寄存器设置运行模式；部分参数也通过寄存器写入电机。

### 反馈模型

RX 路径解包：

- 原始角度
- 原始转速
- 原始扭矩
- 错误状态
- 使能位

然后转换成统一单位并更新多圈角度累计。在线状态由回复到达驱动，不依赖主动上报。

### 配置

`dm,motor` 主要属性包括：

- `tx_id`
- `rx_id`
- `can_channel`
- `controllers`
- `p_max`
- `v_max`
- `t_max`
- `read_only`

`p_max`、`v_max`、`t_max` 同时参与两件事：

- 协议量化时的编码范围。
- `motor_status_t` 中对外暴露的限制范围。

`read_only` 会跳过自动使能和周期控制发送，只保留反馈接收，适合监听场景。

### 边界

- 极限参数不是建议值，而是编码边界；配错会直接影响协议映射。
- 回复超时阈值必须与发送周期一起看，单独缩短超时会造成误报离线。
- `read_only` 节点不应再被当作普通闭环执行器使用。

## MI

### 职责

MI 驱动负责扩展帧 CAN 协议下的模式切换、参数写入和控制发送。它把不同控制语义拆成不同的通信类型，并在模式切换时把控制器参数同步给电机。

### 协议模型

MI 使用扩展 CAN ID，通信类型编码在 ID 中。典型路径包括：

- `MotorEnable` / `MotorStop`
- `SetPosZero`
- `MotionControl_MIT`
- `SetSingleParameter`

其中：

- `MIT` 直接发送位置、速度、增益和扭矩。
- `PV` 通过连续参数写设置速度上限和位置目标。
- `VO` 通过参数写设置电流限制和速度目标。

### 反馈模型

MI 属于请求-回复型。驱动周期发送控制请求并期待反馈：

- 回复到达后刷新在线状态。
- 反馈被解包为角度、速度、扭矩和累计角度。
- `status.enabled` 暴露的是 `requested_enabled`，表示主控的使能意图。

### 配置

`mi,motor` 主要属性包括：

- `id`
- `tx_id`
- `rx_id`
- `can_channel`
- `gear_ratio`
- `controllers`

控制器既参与选择模式，也提供模式相关参数。`gear_ratio` 影响反馈和目标的机械侧含义，修改量纲时应一并核对。

### 边界

- `PV` 和 `VO` 是通过参数写拼出来的高层语义，不是协议原生单帧命令。
- 模式切换和控制发送之间有先后关系，不能把两者折叠成无状态打包。
- `enabled` 的语义偏控制面，不应误当成硬件确认。

## RS

### 职责

RS 驱动服务 Robstride 系列电机，整体结构与 MI 相近，但协议细节、型号参数和自动上报能力不同。它负责：

- MIT 模式打包。
- 使能、停止、清零和清错控制。
- 根据 `auto_report` 决定是否先打开主动上报。

### 协议模型

RS 使用扩展 CAN ID。驱动启动时若 `auto_report` 为真，会先发送报文使电机进入主动上报状态；随后控制循环按请求-回复模型运行。

当前主控制路径集中在 MIT 模式，模式切换也通过参数写完成。

### 反馈模型

反馈由扩展帧回复提供。驱动在收到反馈后：

- 调用 `motor_can_sched_report_rx()`。
- 解析当前位置、速度、扭矩和错误。
- 刷新 `motor_link`。

`status.sum_angle` 目前直接暴露当前角度，不是 DJI/DM 那样的独立多圈累计量，维护者改动前应确认应用是否依赖此差异。

### 配置

`rs,motor` 主要属性包括：

- `id`
- `tx_id`
- `rx_id`
- `motor_type`
- `can_channel`
- `controllers`
- `auto_report`
- `p_max`
- `v_max`
- `t_max`

`motor_type` 体现型号差异，`p_max`、`v_max`、`t_max` 定义量化边界。

### 边界

- `auto_report` 是链路策略，不是控制模式选择。
- 当前实现偏 MIT 导向，扩展 PV/VO 前先确认协议是否支持稳定语义。
- `sum_angle` 语义与其它驱动不完全一致，文档和调用方都应明示。

## LK

### 职责

LK 驱动服务 Ling Kong 电机，负责：

- 标准帧控制命令打包。
- 速度、扭矩、多圈位置目标发送。
- 在需要时把控制器参数写回电机内部参数寄存器。

它更像“协议自带内部控制器，主控负责配置和喂目标”的驱动。

### 协议模型

LK 使用标准帧，通过不同命令字区分：

- 电机启停
- 清错
- 设零
- 速度环
- 扭矩环
- 多圈位置环
- 参数读写

驱动有两条工作路径：

- 周期发送控制命令。
- 独立发送参数更新命令。

这也是 `lk_tx_data_handler()` 和 `lk_tx_params_data_handler()` 分开的原因。

### 反馈模型

LK 同样属于请求-回复型，但控制命令和参数命令都可能触发回复。驱动会：

- 从反馈中更新角度、速度、扭矩和温度。
- 用 `requested_enabled` 表示使能意图。
- 在初始化和运行中按需刷新参数同步标记。

### 配置

`lk,motor` 主要属性包括：

- `id`
- `tx_id`
- `rx_id`
- `can_channel`
- `controllers`

LK 的控制限制更多来自运行时 setpoint：

- 速度模式下用 `torque_limit` 作为电流/扭矩限制输入。
- 位置模式下用 `speed_limit` 作为速度上限输入。

如果应用未显式提供限制，驱动会采用内部默认值。

### 边界

- 控制器在 LK 中主要是参数来源，不是每周期的本地控制计算器。
- 参数更新是独立异步路径，不能在 `motor_set()` 里假定已经立刻生效。
- `SET_ZERO` 使用协议定义的清零命令，改动前要先确认是否影响持久化零点语义。

## VESC

### 职责

VESC 驱动负责：

- 扩展帧控制命令打包。
- 周期发送最新控制目标。
- 周期发送 PING 并接收 PONG。
- 结合状态帧和主动探测维护在线状态。

### 协议模型

VESC 使用扩展 CAN ID。控制目标按当前缓存状态周期发送，不要求每次 `motor_set()` 立刻上总线。驱动支持：

- `VO + TORQUE`
- `VO + SPEED`
- `PV + POSITION`

这几种语义都可直接映射到协议原生命令，因此控制器通常不是必需的；只有应用需要显式枚举或强制选择控制器时才有必要配置 `controllers`。

### 反馈模型

在线状态由两类接收事件共同维护：

- `CAN_PACKET_STATUS` 及其扩展状态帧。
- `CAN_PACKET_PONG`。

后台监视任务定期执行三件事：

- 发送控制帧。
- 发送 PING。
- 检查距最近一次接收的时间，必要时标记离线。

这意味着 VESC 的在线模型本质上是“最近接收时间窗口”，而不是丢失回复计数。

### 配置

`vesc,motor` 主要属性包括：

- `id`
- `rx_id`
- `can_channel`
- `gear_ratio`
- `controllers`
- `kv`
- `pole_pairs`
- `v_max`
- `p_max`
- `t_max`
- `i_max`
- `freq`

其中：

- `rx_id` 是主机在 `PING/PONG` 中使用的 host ID，不是传统意义上的“电机反馈 ID”。
- `freq` 控制主动探测频率，并间接影响离线超时窗口。
- `v_max`、`p_max`、`t_max`、`i_max` 描述控制和量化边界。

### 边界

- `motor_set()` 只是更新最新目标；真正发送由后台周期任务决定。
- 在线恢复必须来自状态帧或 `PONG`，不能因发送控制帧而恢复。
- `freq` 改动会连带影响探测流量和离线判定，不能只当性能参数看。

## Yinshi LA

### 职责

Yinshi LA 驱动位于 `drivers/motor/la_yinshi/`，但它不实现旋转电机 API，而是线性执行器接口 `include/zephyr/drivers/linear_actuator.h`。把它与旋转电机放在同一目录，主要是为了复用子系统组织和遥测习惯，而不是共享控制模型。

### 协议模型

协议运行在 UART 或 RS485 上，不走 CAN。驱动负责：

- 打包自定义二进制命令帧。
- 解析回复帧和状态查询帧。
- 在超时和重试后更新在线状态。

这套协议是同步请求-回复模型，带显式校验和和最大重试次数。

### 反馈模型

驱动维护一份执行器状态缓存，主要包括：

- 目标位置
- 当前位置
- 温度
- 电流
- 推力或内部力值
- 故障标志

在线状态由 UART 收到有效回复或重试超时驱动，离线遥测原因通常是 `RX_TIMEOUT`。

### 配置

`yinshi,la` 主要属性包括：

- `uart`
- `id`
- `de-gpios`
- `max-retries`
- `reply-timeout-ms`

这里的配置决定链路层行为，不涉及 motor 控制器。

### 边界

- 不要把 LA 驱动强行接入 `motor.h`；线性执行器和旋转电机的目标模型不同。
- 它可以复用遥测风格，但不应复用旋转电机的控制器语义。

## 测试视角

`tests/native_sim/motor_driver_sim` 当前重点覆盖：

- DM、MI、RS、LK 的请求-回复在线模型。
- DJI 的周期上报在线模型。
- 各类驱动的连续控制打包顺序。
- 控制发送频率窗口。
- 回复背压下 DJI 控制流不被饿死。

新增驱动或调整现有协议路径时，优先判断它更像哪一类模型，再按对应测试方式补覆盖。最容易出问题的通常不是单帧编码，而是在线恢复、发送节奏和多驱动共享总线时的公平性。
