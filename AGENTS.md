# AGENTS.md

本文件定义仓库内 agent 的工作方式，适用于责任人、维护者、贡献者与评审者。

## 任务目标

优先做能解决当前请求的最小正确改动。

优先采用局部、可复核且低风险的修改；除非任务明确要求，不改变既有代码结构。

## 工作规则

- 编辑前先阅读相关代码。
- 修改范围仅覆盖请求目标行为。
- 不重排、重命名、移动或重构无关代码。
- 不要修改生成文件、忽略文件或本机专属文件，除非任务要求并得到确认。
- 不要修改 `.gitignore`、CI 配置、`west` manifest、board 定义，除非该任务明确授权。
- 不回滚用户修改或仓库中其他未涉本次任务的本地改动。
- 若工作区有未提交变更，提交流程前明确自己的变更边界。
- 优先沿用仓库现有模式、Zephyr API 和现有辅助函数，避免引入新抽象。
- 仅在边界和行为不明显时补充注释。
- 非接口修改场景下保持公共接口稳定。
- 当行为、配置、devicetree binding、公共 API 或维护流程变化时更新文档。

## 代码变更

### 实施前

1. 阅读定义当前行为的相关文件。
2. 锁定可承载本次改动的最小模块边界。
3. 明确需要执行的测试或构建以验证结果。

### 实施中

- 每个补丁保持单一、简单、可复核。
- 不要把整理性改动与行为改动混在同一补丁。
- 不要把策略性决定藏进底层辅助函数。
- 将硬件相关时序、CAN ID、devicetree 属性和 Kconfig 默认值视为公共契约。
- Zephyr 代码中优先使用结构化 API、devicetree helper、Kconfig 依赖关系和子系统约定，避免
临时拼接（ad-hoc）字符串或路径处理。

### 实施后

- 先执行最窄范围、最相关的检查。
- 若触及共享驱动、公共 API、binding、调度逻辑、CI 或测试，补充更广泛校验。
- 记录未执行到的检查及原因。

## 测试与构建

执行方向化验证：

- 本地逻辑或驱动状态改动：运行对应的 `tests/native_sim/*` 测试，使用
  `west twister`。
- 电机公共行为改动：可行时运行 motor 的 `native_sim` 用例。
- 板级/样例改动：执行对应的 `west build -b <board> <sample>`。
- 与格式相关的改动：执行 `git diff --check`。
- CI 流程变更：说明每个受影响 job 的本地复现方式。

常用本地命令：

```shell
west twister -T tests/native_sim/<case> --platform native_sim/native/64 --inline-logs
west twister -T tests/native_sim --platform native_sim/native/64 --inline-logs
west build -b <board> <sample-or-app> --pristine
git diff --check
```

未实际在硬件上验证，不得在报告中声明通过硬件验证。
未经 owner 明确要求，不要刷写板卡、使用 OpenOCD、占用调试探针。

## 提交规范

按逻辑目的组织提交，确保提交颗粒度清晰，便于 review 与回滚。

合理的提交边界：

- 单一行为修复；
- 单一测试补充；
- 单一文档更新；
- 针对本次改动的单一清理改动；
- 单一 CI 或构建系统变更。

避免把不相关的驱动、文档、格式和 CI 变更混合在一个提交中。

使用 Conventional Commit 主题：

```text
fix(motor): 处理请求回复超时
test(motor): 覆盖 DJI 周期上报恢复场景
docs: 更新 native_sim 工作流文档
refactor(chassis): 拆分 board 初始化
ci: 新增 native_sim twister 任务
```

提交说明应解释契约变化，不要叙述执行过程。标题保持简洁，正文补充背景、
约束与验证信息。

提交流程建议：

1. 执行 `git status --short`。
2. 仅暂存本次任务涉及的文件。
3. 复核 `git diff --cached`。
4. 执行或记录相关检查结果。

仅暂存本次任务产生的变更，勿包含任务外的本地预存改动。

## 分支约定

使用规范且清晰的分支名，不要使用 agent 标识前缀。

建议前缀：

- `fix/`
- `feat/`
- `docs/`
- `test/`
- `refactor/`
- `ci/`
- `chore/`

示例：

```text
fix/motor-link-timeout
test/native-sim-motor-state
docs/module-reference
ci/native-sim-twister
```

功能/修复分支应基于预期上游分支，通常是 `master`（除非 owner 指定其他基线）。

## PR 规范

仅在分支包含结构清晰的提交并完成（或明确记录）本地校验后再提交 PR。

PR 标题：

- 使用 Conventional 风格。
- 不要包含 agent 名称或标签。

PR 内容应包含：

- 行为与文档变更摘要。
- 已执行的测试与构建。
- 是否完成硬件验证（如有）。
- 已知限制与后续工作。
- CI、Kconfig、devicetree、公共 API 影响说明。

没有更严格模板时，建议采用如下结构：

```markdown
## 概要

## 测试

## 硬件

## 影响

## 限制
```

保持 PR 小而聚焦。若任务边界扩展到无关工作，拆分分支或先与 owner 对齐。

## CI/CD 跟进

PR 提交后：

1. 关注所有必跑的 CI/CD 检查项。
2. 修改代码前先查清失败日志。
3. 用最小目标提交修复失败。
4. 推送修复并等待 CI/CD 再次执行。
5. 在 CI/CD 全绿之前，不得声称任务完成；若遗留失败明确与本次任务无关且已记录，可注明。

对不稳定或基础设施类失败，如合理可重跑一次；若重复发生，应按真实问题处理。

若无法查看、重跑或等待 CI/CD，必须明确说明限制，并记录已完成的本地校验。
若 owner 未要求 PR，不要为了完成该章节而创建 PR。

## Review 处理

处理 review 时：

- 全量阅读 review 线程后再改。
- 直接修正明确请求的问题。
- 不捆绑无关改进。
- 对每条可执行建议，给出已变更项并逐一回复/关闭。
- 若某请求不安全或超出范围，说明原因并给出更小替代方案。

## 文档管理

文档也是接口的一部分。

当以下内容变化时更新文档：

- 公共 API；
- devicetree binding；
- Kconfig 选项；
- 板级行为；
- CI 或发布流程；
- 测试与仿真行为；
- 样例用法；
- 驱动在线/离线语义；
- CAN 帧调度、优先级、时序、telemetry。

文档应记录可维护契约，不记录过程叙事、营销化描述或临时 debug 备注。

## 安全要求

- 未经明确要求，不执行破坏性 git 命令。
- 不因文件未跟踪就直接删除本地文件。
- 不默认提交忽略文件中的内容。
- 不泄露密钥、私有路径、访问令牌或仅本机可复现的构建产物。
- 未实际执行的测试、构建、刷写、PR 操作，不得写入已完成。

无法确认时，优先保留用户已有改动，并选择可回滚、影响最小的操作。
